*"3개월 개발기" 2편. 3명이 Gemini·Claude Opus·Cursor·Claude Code 를
프로젝트 단계별로 어떻게 분업했는지 — 설계-도구 페어링 다섯 개, 각
단계에서 실제로 드러난 장면들, 그리고 왜 구현 단계에서 Claude Code
가 결정적이었는지.*

## 단계-도구 페어링이라는 프레임

Ep 1 에서 짚은 제약 — 3명, 1 GPU, 제한된 자원 — 이 아키텍처 선택만
강제한 게 아니라 *AI 협업 자체의 조직 방식* 도 강제했다. "Claude 한
개 구독으로 모든 단계를 커버" 식의 접근은 초반 며칠 만에 좌초한다.
각 단계마다 다른 AI 의 다른 강점이 필요했고, 도구를 의도적으로 분리
배치하는 게 속도와 품질 양쪽에서 더 나았다.

5단계로 정리된 흐름은 아래와 같다. 순서대로 설명한다.

## Phase A — 아이디에이션 (Gemini)

초기 컨셉 탐색과 브레인스토밍은 Gemini 로 했다. ALS 대체 옵션,
Black-Litterman 탐색, 모델 앙상블 비교 — 아키텍처 후보를 *넓게 훑는*
작업이 Gemini 의 광범위한 지식 베이스와 맞았다.

가장 큰 가치가 나온 건 *학제간 피처 아이디어 탐색* 이었다.
"화학 반응속도론으로 소비 행동을 설명할 수 있는가?", "상품 채택이
전염병 확산과 구조적으로 동등한가?" 같은 질문을 던지면서, 금융
고객 행동과 구조적으로 동형인 문제를 다른 학문이 어디서 이미 풀었는지
체계적으로 뒤졌다. PM 이 FRM · 신용분석 도메인 전문성을 제공하고,
Gemini 가 학제간 연결 고리를 제안하는 분업.

이 과정에서 *"구조적 동형사상 (structural isomorphism)"* 이라는 핵심
개념이 부상했다. 11개 학문 분야에서 피처를 가져오겠다는 설계 방향이
여기서 확정됐고, 이후 모든 기술적 결정의 근간이 됐다. 어떤 깊은 기술
선택이 아니라 *"어떤 분야에서 비슷한 문제를 이미 풀었는가"* 를 빠르게
스캔하는 데 Gemini 가 최적이었다.

## Phase B — 기술 검증 (Claude Opus)

아이디어를 아키텍처로 구체화하는 단계는 Claude Opus 였다. 수학적
엄밀성이 필요한 loss function 설계, 데이터 리키지 검증, 정규화
파이프라인 — 기술적 깊이가 필요한 작업이 여기 모였다.

전문가 후보를 하나씩 실현 가능성 검증했다. "HGCN 이 MCC 계층 구조에서
작동하는가?", "Mamba 가 17개월 시퀀스에 충분히 효율적인가?" 같은
질문에 대해 Opus 와 깊이 있는 기술 토론. PLE vs MMoE 트레이드오프
분석, adaTT 의 loss-level vs representation-level 전이 비교도 이
단계에서 이루어졌다.

Opus 는 *가정을 도전하는 역할* 에서도 중요했다. "Black-Litterman 이
정말 적합한가?" 라는 반론을 Opus 가 제기해서 PLE 로의 전환이 촉진됐다.
동질적 MoE 에서 발생하는 *expert collapse* 문제를 Opus 와의 대화에서
먼저 식별했고, 이게 Phase E 에서 NeurIPS 2024 sigmoid gate 논문으로
이어지는 실마리가 된다 — 그리고 그 결론은 한참 뒤에 다시 뒤집히게
되는데, 이것도 결국 같은 맥락의 후속편이 된다.

Phase B 의 산출물은 19편의 기술 참조 문서 (`.typ` 파일). 이게 이후
구현 단계에서 각 AI 에이전트가 참조하는 *설계 명세서* 역할을 했다.

## Phase C — 코드 환경 정비 (Cursor)

GitHub 기반 코드 환경 구성, 프로젝트 구조 설계, 초기 보일러플레이트
생성은 Cursor 로 수행했다. IDE 통합 환경에서의 빠른 네비게이션과
리팩토링이 Cursor 의 강점.

이 단계의 가장 중요한 산출물은 6편의 초기 설계 문서 (00-09 아키텍처
명세) 와 `CLAUDE.md` 가드레일 수립이었다. Config-driven 원칙, 관심사
분리, 리키지 방지 규칙 — 이후 모든 AI 에이전트가 따라야 할 "헌법" 이
*코드 한 줄 작성되기 전에* 확립됐다. 순서가 의도적이었다: 가드레일이
먼저, 에이전트 실행이 그다음. 이 순서를 반대로 했다면 Phase D 의
병렬 구현이 통합 지점에서 깨졌을 것이다.

## Phase D — 병렬 구현 (Claude Code · Opus/Sonnet)

본격 구현 단계에서 각 팀원이 *AI 에이전트의 "팀장"* 역할을 맡았다.
Claude Code 환경에서 Opus 와 Sonnet 을 병렬로 굴려 서로 다른 모듈을
동시에 구현. 3명의 인간이 각각 AI 에이전트 팀을 하나씩 이끌었다.

- **PM / 리드 아키텍트의 AI 팀** — Opus 에 아키텍처 수준 결정 (PLE
  config, adaTT 태스크 그룹, 로짓 전이 설계), Sonnet 에 빠른 코드
  구현 (generator, adapter, pipeline runner). 이 팀에서 label leakage
  3건 연쇄 탐지, FP16 NaN 원인 4가지 진단, GPU 활용률 최적화 같은
  핵심 디버깅 세션이 나왔다.
- **엔지니어 1 의 AI 팀** — 데이터 수집 파이프라인, HIVE 병렬 쿼리
  로직, 10개 generator (TDA, HMM, Mamba, Graph, GMM 등) 의 피처
  엔지니어링, 피처-비즈니스 역매핑 레지스트리 담당.
- **엔지니어 2 의 AI 팀** — 모델 학습, 수학적 검증, 지식 증류 (PLE
  → LGBM) 담당.

3개 팀이 병렬로 굴러가면서 일관성을 유지할 수 있었던 건 Phase C 의
`CLAUDE.md` 가드레일과 *인터페이스 계약 검증 프로세스* 덕이었다.
파일 A 가 저장하는 키 이름과 파일 B 가 읽는 키 이름이 일치하는지 —
매 병렬 작업 후 반드시 검증. 이 루틴 없이 병렬 AI 에이전트를 돌리면
통합 단계에서 미묘한 키 네이밍 불일치로 런타임 실패가 누적된다.

## Phase E — 실험 + 논문 (Claude Code Extension)

ablation 실험 단계에서는 Claude Code 를 *실시간 모니터링 도구* 로
썼다. 진행 상황, GPU 활용률, 에러 탐지를 실시간으로 보면서 조정.
이 중에 PLE toggle 버그를 라이브 디버깅으로 발견했다 — `use_ple=false`
설정이 전문가 구성 자체를 바꿔버려서 공정한 비교가 성립하지 않던
문제. 실험 돌리면서 모니터링하지 않았으면 며칠 분량의 결과가
무효화됐을 것이다.

실험 대기 시간을 활용한 문헌 조사도 이 단계의 특징이다. PLE val_loss
가 수렴하지 않는 현상을 관찰하면서 Opus 와의 대화로 *"softmax gate
의 경쟁적 특성이 이종 전문가 간 수렴을 방해한다"* 는 가설을 세웠고,
NeurIPS 2024 의 sigmoid gate 논문을 찾아 이론적 근거를 확보, 그 자리
에서 sigmoid gate 을 구현했다. 실험 결과도 일치했다 — sigmoid 가
softmax 를 일관되게 앞섰다. 이 발견은 그때 결론으로 문서화됐고,
이후 실험 설계에도 반영됐다.

이야기가 여기서 끝났다면 Phase B 의 expert collapse 실마리가 Phase E
에서 깔끔히 결실을 맺은 서사가 됐을 것이다. 실제로는 그렇지 않았다.
몇 주 뒤 uncertainty weighting 의 구현 버그를 수정하면서 결과가
역전된다. 올바른 손실 가중치 적용 후 softmax 가 NDCG 메트릭에서
sigmoid 를 앞서기 시작했다. 돌아가 보니 근본 원인은 깨진 손실 가중치
였다 — 13개 태스크가 모두 동일하게 가중될 때는 수가 많은 binary 분류
그래디언트가 multiclass · regression 을 압도했고, sigmoid 의 비경쟁적
라우팅이 그 조건에서 우연히 방화벽 역할을 했다. 손실 가중치가 제대로
적용되자 이번엔 softmax 의 경쟁적 라우팅이 태스크 유형 간 그래디언트
오염을 막는 구조적 장벽으로 작동했다. 13개 태스크 · 3개 유형의 *이질적
MTL* 에서는 문헌의 "sigmoid 가 낫다" 결과가 곧바로 전이되지 않는다.

교훈은 sigmoid vs softmax 의 승패가 아니라, *훈련 환경의 버그가
아키텍처 결론을 오염시킬 수 있다* 는 방법론적 지점이다. "sigmoid 가
낫다" 는 결론은 깨진 환경에 대한 유효한 적응이었지, 아키텍처적
우위가 아니었다. 근본 원인 조사 없이는 그 결론이 무기한 이어졌을
것이다. 이 재조사가 가능했던 것 역시 장기 컨텍스트 덕분이다 — 수 주
전의 sigmoid 채택 맥락이 남아있는 상태에서 새 증거 (uncertainty fix
이후 성능 역전) 를 만나 이전 결론을 재검토하는 흐름.

논문 4편 (영문/한국어) 과 22편의 기술 문서가 Claude 와의 반복 작업으로
생성·다듬어졌다. 이 단계의 AI 는 텍스트 생성기가 아니라 프로젝트의
의미를 함께 구성하는 *사고 파트너* 였다.

## 왜 Claude Code 였는가

5개 도구를 나눠 썼지만, 구현·실험·논문 단계에서 Claude Code 를
*대체 불가능하게* 만든 세 가지 요소가 있었다.

**1M context 와 세션 내 연속 추적.** label leakage 3건의 연쇄 추적은
며칠에 걸친 작업의 맥락을 유지해야만 가능했다. 첫 번째 (has\_nba 중복
컬럼) 수정 후 같은 세션에서 두 번째 (ground truth glob 정렬), 세 번째
(generator label 입력) 를 발견할 수 있었던 건 이전 수정의 맥락이
살아있었기 때문이다.

**전역 조망 + 국소 추적의 동시 수행.** FP16 NaN 4개 (CGC entropy, OT
Sinkhorn, Causal DAG, logits) 의 동시 진단은 모델 아키텍처 전체를
한 번에 조망하면서 각 expert 의 수치 연산을 추적해야 했다. 파일
하나씩 순회하는 방식으로는 불가능한 작업.

**관찰-가설-문헌-구현, 그리고 그 결론의 재검토.** 위에서 언급한
sigmoid ↔ softmax 아크가 이 역량의 가장 좋은 예다. 실험 분석 → 가설
→ 문헌 → 구현이 한 세션 안에서 이어졌고, 그 후 몇 주 뒤 새 증거가
나왔을 때 *이전 결론을 기억한 상태* 에서 재조사에 들어갈 수 있었다.
도구를 바꿔 탔다면 "예전에 sigmoid 를 택한 이유" 자체를 재구축하며
시간을 썼을 가능성이 높다.

## 굳은 패턴 몇 가지

프로젝트 전 과정에서 의도적으로 설계하지 않았는데 자연스럽게 드러난
패턴들. 우리 팀이 자주 입에 올렸던 것 순으로.

**AI 는 HOW, 인간은 WHAT 과 WHY 를 결정한다.** AI 가 코드와 텍스트를
생성하지만 아키텍처 결정은 인간이 내린다. "구조적 동형사상" 이라는
핵심 통찰은 인간-AI 대화에서 발현됐지만, 그것을 *설계 원칙으로 채택
한 것* 은 인간의 판단이었다.

**에이전트 전에 가드레일을 세운다.** `CLAUDE.md` 는 코드가 아니라
코드 *이전에* 작성됐다. 헌법이 먼저이고 입법이 나중인 것처럼.

**이종 AI = 이종 전문가.** 모델의 이종 전문가 설계 철학이 개발 도구
선택에도 그대로 적용됐다. 각 AI 도구가 특화된 역할을 수행하면서,
단일 도구로는 도달할 수 없는 품질·속도를 확보했다. 이게 Phase A-E
분업의 본질이다.

**AI 로 빠르게 실패한다.** leakage, FP16 NaN, ablation 필터 미작동 —
수동으로는 수일 걸릴 버그를 AI 에이전트와 함께 분 단위로 탐지·수정
했다. 빠른 실패가 빠른 학습으로 이어졌다.
