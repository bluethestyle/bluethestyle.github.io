*"3개월간의 금융 AI 개발기" 2편. 3명의 팀원이 Gemini, Claude Opus, Cursor, Claude Code를 프로젝트 단계별로 어떻게 철저히 분업시켰는지 다룬다. 5단계의 설계-도구 페어링, 각 단계에서 실제로 펼쳐진 장면들, 그리고 특히 구현 단계에서 Claude Code가 왜 그토록 결정적인 역할을 했는지 살펴본다.*

## 단계별 도구 페어링(Pairing) 전략

Ep 1에서 짚었던 가혹한 제약 조건(3명의 인원, 단 1대의 소비자용 GPU, 턱없이 부족한 예산)은 단순히 아키텍처의 선택만을 강제한 것이 아니었다. **AI와의 협업을 조직하는 방식 자체**마저도 철저히 재편하도록 만들었다. "Claude 구독권 하나 끊어서 모든 단계를 커버하자"는 식의 안이한 접근은 프로젝트 시작 며칠 만에 보기 좋게 좌초했다. 각 개발 단계마다 요구되는 AI의 강점이 명확히 달랐기 때문이다. 목적에 맞춰 여러 AI 도구를 의도적으로 분리하고 배치하는 전략이 결과적으로 개발 속도와 결과물의 품질 양쪽 모두에서 훨씬 뛰어난 성과를 냈다.

우리가 프로젝트를 진행하며 정립한 5단계의 협업 흐름은 다음과 같다.

## Phase A — 아이디에이션 (Gemini)

초기 콘셉트 탐색과 브레인스토밍은 전적으로 Gemini에게 맡겼다. 과거의 ALS 모델을 대체할 대안 찾기, Black-Litterman 모델의 적용 가능성 탐색, 앙상블(Ensemble) 기법 비교 등 아키텍처 후보군을 **넓고 얕게 훑어보는** 작업에는 Gemini의 방대한 지식 베이스가 안성맞춤이었다.

하지만 Gemini가 창출한 가장 큰 가치는 바로 **학제 간(Interdisciplinary) 피처 아이디어 탐색**에 있었다. "화학의 반응 속도론(Kinetics) 공식으로 고객의 소비 행동 변화를 설명할 수 있을까?", "특정 금융 상품이 채택되는 과정이 전염병의 확산 모델과 구조적으로 동일하지 않을까?"와 같은 엉뚱하지만 날카로운 질문들을 끊임없이 던졌다. 금융 고객의 행동 패턴과 본질적으로 동일한 문제를 다른 학문 분야에서는 이미 어떻게 풀어냈는지 체계적으로 뒤지기 시작했다. PM이 재무 리스크 관리(FRM)와 신용 분석이라는 도메인 전문성을 제공하면, Gemini가 그와 연관된 다른 학문과의 연결 고리를 척척 제안하는 완벽한 분업이었다.

바로 이 과정에서 **"구조적 동형사상(Structural Isomorphism)"**이라는 핵심 개념이 수면 위로 떠올랐다. 무려 11개의 서로 다른 학문 분야에서 수학적 구조를 차용해 오겠다는 혁신적인 설계 방향이 이때 확정되었고, 이는 이후 진행된 모든 기술적 의사결정의 든든한 근간이 되었다. 이 단계에서 Gemini가 빛을 발한 것은 깊이 있는 기술적 증명 능력이 아니었다. **'어느 분야에서 이와 비슷한 문제를 이미 훌륭하게 풀어냈는가'**를 누구보다 빠르게 훑어내는 탁월한 직관력에 있었다.

## Phase B — 기술 검증 (Claude Opus)

Gemini가 던져준 아이디어를 실제 작동하는 아키텍처로 구체화하는 일은 Claude Opus의 몫이었다. 수학적 엄밀성이 절실히 요구되는 손실 함수(Loss Function) 설계, 데이터 리키지(Data Leakage) 검증, 정규화 파이프라인 구축 등 **깊이 있는 기술적 역량**이 필요한 골치 아픈 작업들이 모두 이 단계에 몰려 있었다.

우리는 선별된 전문가 네트워크 후보들을 하나씩 도마 위에 올리고 실현 가능성을 철저히 검증했다. "HGCN(Hyperbolic Graph Convolutional Network)이 신용카드 가맹점(MCC)의 계층 구조에서도 제대로 작동할까?", "Mamba 아키텍처가 17개월치 시계열 데이터를 처리할 만큼 충분히 효율적일까?"와 같은 고난도 질문들을 두고 Opus와 치열한 기술 토론을 벌였다. PLE와 MMoE 아키텍처 간의 트레이드오프 분석, adaTT 모델의 손실 수준(Loss-level) 전이와 표현 수준(Representation-level) 전이의 비교 역시 모두 이 단계에서 이루어졌다.

무엇보다 Opus는 우리의 **기본 가정에 날카로운 의문을 던지는 역할**을 훌륭히 수행했다. "정말 Black-Litterman 모델이 최선의 선택인가?"라는 본질적인 물음을 Opus가 먼저 꺼내 들었고, 이는 결국 전체 아키텍처를 PLE 구조로 전면 수정하게 만드는 결정적인 계기가 되었다. 동질적인 전문가 네트워크(MoE) 환경에서 흔히 발생하는 '전문가 붕괴(Expert Collapse)' 문제 역시 Opus와의 대화 도중 가장 먼저 제기되었다. 이 논의는 훗날 Phase E 단계에서 NeurIPS 2024에 발표된 Sigmoid Gate 관련 논문으로 이어지는 귀중한 실마리가 된다. (물론 이 결론조차도 한참 뒤에 극적으로 뒤집히게 되는데, 이 흥미로운 비하인드 스토리는 후반부에 다시 자세히 다룬다.)

Phase B를 거치며 우리는 총 19편에 달하는 촘촘한 기술 참조 문서(.typ 파일)를 얻어냈다. 이 문서들은 이후 이어지는 실제 구현 단계에서 각 AI 에이전트들이 헤매지 않도록 방향을 잡아주는 든든한 **설계 명세서** 역할을 완벽히 수행했다.

## Phase C — 코드 환경 정비 (Cursor)

GitHub 기반의 코드 리포지토리 환경 구성, 전체 프로젝트 디렉터리 구조 설계, 그리고 초기 보일러플레이트(Boilerplate) 코드 생성 작업은 모두 Cursor IDE에 일임했다. 개발 환경과 찰떡같이 통합되어 파일 간을 넘나드는 빠른 내비게이션과 대규모 리팩토링 기능은 Cursor가 가진 최고의 강점이었다.

이 단계에서 얻어낸 가장 핵심적인 산출물은 6편의 초기 아키텍처 설계 문서(00~09 아키텍처 명세)와, 프로젝트 전체의 규약집 역할을 하는 CLAUDE.md 가드레일 규칙이었다. 하드코딩을 배제하는 Config-driven 원칙, 엄격한 관심사 분리(Separation of Concerns), 그리고 데이터 리키지 방지 규칙 등 이후 프로젝트에 투입될 모든 AI 에이전트가 예외 없이 따라야 할 기본 규약이 **단 한 줄의 실제 코드가 작성되기도 전에** 굳건히 확립되었다.

이 순서는 철저히 의도된 것이었다. 튼튼한 가드레일을 먼저 세우고, 그 안에서 에이전트들을 뛰놀게 해야만 했다. 만약 이 순서를 반대로 하여 무작정 코딩부터 시작했다면, 다음 단계(Phase D)에서 진행된 3명의 병렬 구현 작업은 최종 통합 단계에서 산산조각이 나고 말았을 것이다.

## Phase D — 병렬 구현 (Claude Code · Opus/Sonnet)

본격적인 코드 구현 단계에 접어들자, 3명의 인간 팀원은 각자 **AI 에이전트들을 거느린 "팀장"**으로 변신했다. CLI 환경인 Claude Code 안에서 똑똑하지만 느린 Opus와, 빠르고 경쾌한 Sonnet을 동시에 병렬로 굴리며 각기 다른 모듈을 동시다발적으로 구현해 나갔다.

- **PM / 리드 아키텍트의 AI 팀:** 큰 그림을 그리는 아키텍처 수준의 의사결정(PLE Config 구조, adaTT 태스크 그룹화, 로짓 전이 설계 등)은 Opus에게 맡기고, 이를 바탕으로 실제 코드를 빠르게 찍어내는 작업(Generator, Adapter, Pipeline Runner 등)은 Sonnet에게 전담시켰다. 레이블 리키지(Label Leakage) 3건 연쇄 탐지, FP16 정밀도 연산에서 발생한 NaN 에러의 원인 진단, GPU 메모리 활용률 최적화 같은 굵직한 디버깅 성과들이 모두 이 팀에서 터져 나왔다.
- **엔지니어 1의 AI 팀:** 방대한 데이터 수집 파이프라인 구축, HIVE 네트워크 병렬 쿼리 로직 설계, 그리고 10개에 달하는 이종 전문가(TDA, HMM, Mamba, Graph, GMM 등)의 복잡한 피처 엔지니어링과 피처-비즈니스 역매핑(Reverse Mapping) 레지스트리 관리를 훌륭하게 소화해 냈다.
- **엔지니어 2의 AI 팀:** 완성된 파이프라인을 바탕으로 한 실제 모델 학습 로직, 복잡한 수학적 검증 과정, 그리고 무거운 PLE 모델을 가벼운 LGBM으로 압축하는 지식 증류(Knowledge Distillation) 파트를 전담했다.

이처럼 3개의 독립적인 팀이 맹렬하게 병렬로 코드를 쏟아내면서도 전체 시스템의 일관성을 완벽하게 유지할 수 있었던 비결은 앞선 Phase C에서 꼼꼼하게 세워둔 CLAUDE.md 가드레일과 **인터페이스 계약 검증 프로세스** 덕분이었다. 파일 A가 저장할 때 사용한 딕셔너리의 키(Key) 이름과, 파일 B가 이를 읽어 들일 때 기대하는 키 이름이 정확히 일치하는지 매 병렬 작업이 끝날 때마다 AI를 통해 강박적으로 검증했다. 만약 이 지루한 확인 루틴 없이 병렬로 AI 에이전트들을 폭주하게 내버려 두었다면, 최종 통합 단계에서 사소한 변수명 불일치로 인한 런타임 에러가 산더미처럼 쌓여 프로젝트 전체가 붕괴했을 것이다.

## Phase E — 실험 및 논문 작성 (Claude Code Extension)

본격적인 제거 실험(Ablation Study) 단계에 돌입하자, 우리는 Claude Code를 훌륭한 **실시간 모니터링 도구**로 적극 활용했다. 터미널에 찍히는 학습 진행 상황, 요동치는 GPU 활용률, 불쑥 튀어나오는 에러 로그를 AI와 함께 실시간으로 지켜보며 즉각적으로 대응했다. 이 과정에서 치명적인 PLE 토글(Toggle) 버그를 라이브 디버깅으로 잡아내는 성과를 올렸다. use_ple=false로 설정할 경우 베이스라인 모델의 전문가 구성 자체가 아예 달라져 버려서, 애초에 공정한 성능 비교가 성립할 수 없었던 황당한 논리적 오류였다. 만약 AI와 함께 실시간으로 로그를 뜯어보며 모니터링하지 않았다면, 귀한 GPU를 며칠 내내 헛돌리며 얻어낸 소중한 실험 결과들이 통째로 휴지 조각이 될 뻔했다.

길고 지루한 모델 학습 대기 시간을 알차게 활용한 밀착 문헌 조사 역시 이 단계의 빼놓을 수 없는 묘미였다. PLE 아키텍처의 검증 손실(Validation Loss)이 좀처럼 수렴하지 않고 요동치는 현상을 지켜보던 중, Opus와의 깊은 대화를 통해 *"Softmax 게이트 특유의 배타적이고 경쟁적인 특성이, 본질적으로 성격이 다른 이종 전문가들 간의 조화로운 수렴을 오히려 방해하고 있는 것 아닐까?"*라는 그럴듯한 가설을 세우게 되었다. 곧바로 문헌을 뒤져 NeurIPS 2024에 발표된 Sigmoid 게이트 관련 논문을 찾아내 탄탄한 이론적 근거를 확보했고, 그 자리에서 즉시 코드를 수정해 Sigmoid 게이트를 시스템에 이식했다. 놀랍게도 실험 결과는 가설과 정확히 일치했다. Sigmoid가 기존의 Softmax 성능을 일관되게 훌쩍 뛰어넘은 것이다. 이 짜릿한 발견은 즉시 공식적인 결론으로 문서화되었고, 이후 진행된 모든 실험 설계의 새로운 기준점이 되었다.

만약 우리의 개발기가 여기서 훈훈하게 마무리되었다면, 앞선 Phase B에서 넌지시 짚었던 전문가 붕괴(Expert Collapse)에 대한 우려가 Phase E에 이르러 Sigmoid 게이트 도입이라는 멋진 해결책으로 완벽하게 결실을 맺은 아름다운 서사로 남았을 것이다. 하지만 현실의 개발은 그렇게 호락호락하지 않았다.

몇 주 뒤, 불확실성 가중치(Uncertainty Weighting) 로직에 숨어 있던 미묘한 구현 버그를 찾아내 수정하자 충격적인 결과가 나타났다. 성능의 우위가 완전히 역전되어 버린 것이다. 올바른 손실 가중치가 제대로 적용되자, 버려졌던 Softmax가 NDCG(랭킹 품질) 메트릭에서 보란 듯이 Sigmoid를 압도하기 시작했다. 당황하여 원인을 파고들어 보니, 근본적인 범인은 이전까지 방치되어 있던 **'깨진 손실 가중치'** 환경 그 자체였다. 성격이 판이한 13개의 태스크가 멍청하게 모두 똑같은 가중치로 묶여 있다 보니, 숫자가 압도적으로 많은 이진 분류(Binary Classification) 태스크의 거대한 그래디언트가 다중 분류나 회귀(Regression) 태스크의 미세한 신호들을 무자비하게 짓밟고 있었던 것이다. 이런 난장판 속에서는 각 전문가가 독자적으로 반응할 수 있게 해주는 Sigmoid의 비경쟁적 라우팅 방식이 우연찮게 훌륭한 방화벽 역할을 해 주었던 것뿐이다. 

하지만 버그가 수정되어 태스크별 손실 가중치가 올바른 균형을 되찾자 상황이 180도 달라졌다. 이제는 오히려 Softmax의 치열한 경쟁적 라우팅 방식이, 성격이 다른 태스크 유형 간에 그래디언트가 마구잡이로 섞여 오염되는 것을 막아주는 든든한 구조적 장벽으로 작동하기 시작한 것이다. 13개의 태스크와 3가지의 완전히 다른 문제 유형이 혼재된 **극도로 이질적인 다중 태스크 학습(MTL)** 환경에서는, 그저 잘 빠진 최신 논문의 "Sigmoid가 대체로 더 낫더라"라는 결론을 맹목적으로 가져다 쓸 수 없다는 뼈아픈 교훈을 얻는 순간이었다.

이 드라마틱한 에피소드가 우리에게 남긴 진짜 교훈은 '결국 Sigmoid가 이겼냐, Softmax가 이겼냐' 하는 얄팍한 승패가 아니다. **훈련 환경에 숨어 있는 아주 작은 버그 하나가, 거대한 아키텍처 설계에 대한 결론 자체를 완전히 오염시켜 버릴 수 있다**는 소름 돋는 사실이다. "Sigmoid가 낫다"라는 이전의 결론은, 사실 버그로 인해 망가져 있던 당시 환경에서 모델이 살아남기 위해 찾아낸 '유효한 적응'이었을 뿐, 아키텍처 자체의 절대적인 우위를 의미하는 것이 아니었다. 만약 우리가 근본 원인을 집요하게 파고들지 않았다면, 그 잘못된 결론은 아무런 의심 없이 정답으로 굳어져 버렸을 것이다. 

이처럼 과거의 결론을 주저 없이 다시 도마 위에 올리고 재조사할 수 있었던 것은, Claude Code가 제공하는 **압도적으로 긴 장기 컨텍스트(Long Context)** 덕분이었다. 수 주 전에 우리가 왜 Sigmoid를 선택했는지에 대한 치열했던 고민과 맥락이 세션 기록 속에 고스란히 살아 숨 쉬고 있었기 때문에, 버그 수정 이후 성능이 역전되었다는 새로운 증거를 마주했을 때 과거의 결론을 자연스럽게, 그리고 즉각적으로 다시 들여다볼 수 있었다.

프로젝트의 대미를 장식한 4편의 논문(영문/한국어)과 무려 22편에 달하는 묵직한 기술 문서들 역시, Claude와의 끊임없는 대화를 통해 초안이 작성되고 정교하게 다듬어졌다. 이 마지막 단계에 이르러 우리에게 AI는 단순한 텍스트 생성기나 코드 자동 완성 도구가 아니었다. 프로젝트가 가진 진짜 의미를 함께 고민하고 구성해 나가는 진정한 의미의 **사고 파트너(Thinking Partner)**였다.

## 수많은 도구 중 왜 하필 Claude Code였는가

프로젝트 전반에 걸쳐 5가지 이상의 다양한 AI 도구를 목적에 맞게 쪼개어 사용했지만, 특히 구현, 실험, 논문 작성이라는 핵심 단계에서 **Claude Code를 절대 대체 불가능한 존재**로 만들어준 데에는 세 가지 압도적인 요소가 있었다.

**첫째, 한계를 모르는 긴 컨텍스트와 세션 내 연속적인 추적 능력이다.** 우리가 골치 아픈 레이블 리키지(Label Leakage) 3건을 꼬리에 꼬리를 물고 연달아 잡아낼 수 있었던 것은, 며칠 치에 달하는 방대한 작업 맥락이 에이전트의 기억 속에 고스란히 살아있었기 때문이다. 첫 번째 원인(has_nba 중복 컬럼)을 찾아내어 수정한 바로 그 세션 안에서, 곧이어 두 번째 원인(Ground Truth Glob 파일의 정렬 순서 오류), 그리고 세 번째 원인(Generator의 레이블 입력 단계 오류)까지 내리 짚어낼 수 있었다. 이전의 수정 내역과 논리적 흐름이 기억 속에 꽉 잡혀 있었기에, 다음에 파고들어야 할 단서가 자연스럽게 눈에 들어온 것이다.

**둘째, 숲 전체를 조망하면서도 각 나무의 디테일을 동시에 추적하는 시야다.** FP16 정밀도 환경에서 동시다발적으로 터져 나온 4개의 골치 아픈 NaN 에러(CGC Entropy, OT Sinkhorn 연산, Causal DAG 노드, 최종 Logits 값)를 한꺼번에 진단해야 했을 때, 우리는 거대한 모델 아키텍처 전체를 넓게 조망하면서도 동시에 각 전문가 모듈 깊숙한 곳의 수치 연산 흐름을 놓치지 않고 따라가야만 했다. 이는 인간 엔지니어가 IDE 창을 띄워놓고 파일 하나하나를 일일이 열어보는 고전적인 방식으로는 도저히 해낼 수 없는 묘기였다.

**셋째, 현상의 관찰부터 가설 수립, 논문 탐색, 실제 코드 구현, 그리고 과거 결론의 쿨한 재검토까지 이어지는 매끄러운 흐름이다.** 앞서 길게 설명했던 Sigmoid와 Softmax 사이의 엎치락뒤치락하는 성능 역전 서사가 이 역량을 가장 눈부시게 보여준다. 실험 결과 분석 → 가설 수립 → 관련 문헌 탐색 → 코드 구현에 이르는 복잡한 사이클이 단 하나의 세션 안에서 끊김 없이 이어졌다. 무엇보다 몇 주 뒤 새로운 반박 증거가 튀어나왔을 때도, **과거에 우리가 어떤 치열한 논리를 거쳐 이전 결론에 도달했었는지가 고스란히 남아 있는 상태**였기에 망설임 없이 즉각적인 재검토에 돌입할 수 있었다. 만약 중간에 다른 도구로 갈아탔거나 맥락이 끊겼다면, 우리는 "아, 그때 우리가 도대체 왜 Softmax를 버리고 Sigmoid를 택했더라?"라며 지난 기억을 더듬고 문서들을 뒤적이는 데에만 귀한 며칠을 허비했을 것이다.

## 의도치 않게, 그러나 자연스럽게 드러난 협업의 패턴들

프로젝트 전 과정에 걸쳐, 애초에 억지로 기획하거나 의도적으로 설계하지 않았음에도 불구하고 너무나 자연스럽게 자리 잡은 훌륭한 협업 패턴들이 있었다. 우리 팀원들이 회의 시간마다 가장 자주 입에 올렸던 순서대로 정리해 본다.

**첫째, AI는 '어떻게(HOW)'를 고민하고, 인간은 '무엇을(WHAT)' 그리고 '왜(WHY)' 할 것인지를 결정한다.** 막막한 빈 화면에 숨결을 불어넣어 실제 작동하는 코드와 유려한 텍스트를 쉼 없이 쏟아내는 것은 AI의 몫이었다. 하지만 수많은 선택지 중에서 어떤 아키텍처를 취할 것인지 최종적인 결단을 내리는 것은 철저히 인간의 영역이었다. "구조적 동형사상"이라는 반짝이는 통찰은 분명 인간과 AI의 핑퐁 같은 대화 속에서 빚어졌지만, 그것을 흔들리지 않는 **프로젝트의 핵심 설계 원칙으로 채택하고 못 박은 것**은 결국 인간의 단호한 판단이었다.

**둘째, 에이전트를 풀어놓기 전에 반드시 튼튼한 가드레일부터 세워야 한다.** 우리 프로젝트의 규약집인 CLAUDE.md 문서는 코드가 작성된 후에 부랴부랴 끼워 맞춘 것이 아니라, 단 한 줄의 코드가 쓰이기도 **전에** 가장 먼저 작성되었다. 흔들리지 않는 규약과 원칙이 먼저 확립되어야, 비로소 그 위에서 안전하고 빠른 구현이 가능하다는 진리다.

**셋째, '이종 AI'의 결합은 곧 '이종 전문가'의 결합과 같다.** 우리 추천 모델의 뼈대를 이루는 '서로 다른 이종 전문가들의 조화로운 결합'이라는 설계 철학은, 역설적이게도 우리가 AI 개발 도구들을 선택하고 조합하는 방식에도 그대로 투영되었다. 각각의 AI 도구(Gemini, Opus, Cursor, Claude Code)가 자신이 가장 잘하는 특화된 역할에만 집중하게 만들자, 단일 도구 하나만 쥐고 끙끙거렸을 때는 도저히 상상조차 할 수 없었던 압도적인 퀄리티와 개발 속도를 확보할 수 있었다. 이것이 바로 Phase A부터 E까지 이어졌던 철저한 도구 분업 전략의 본질이다.

**넷째, AI와 함께라면 그 누구보다 '빠르게 실패할 수' 있다.** 피를 말리는 데이터 리키지, 원인을 알 수 없는 FP16 정밀도 환경에서의 NaN 에러, 제거 실험(Ablation) 필터의 논리적 오작동까지. 만약 인간 엔지니어가 수동으로 로그를 뒤지며 디버깅했다면 꼬박 며칠 밤을 새웠을 법한 악랄한 버그들을, 우리는 AI 에이전트와 머리를 맞대고 불과 몇 분 단위로 찾아내어 박살 냈다. 두려움 없는 '빠른 실패(Fast Fail)'는 곧 프로젝트 전체의 '빠른 학습(Fast Learning)'으로 직결되었다.

## 다음 편

이어지는 **에피소드 3**에서는 Phase D 단계에서 3개의 AI 팀이 무섭게 병렬로 굴러가면서도 프로젝트의 일관성이 산산조각 나지 않도록 단단히 붙잡아 주었던 **구체적이고 실제적인 장치들**을 들여다볼 것이다. CLAUDE.md 가드레일에 적혀 있던 구체적인 조항들, 8개의 콘텍스트 파일로 구성된 초기 메모리 뱅크(Memory Bank) 시스템, .claude/RULES.md와 .cursorrules 설정 파일 간의 눈물겨운 동기화 과정, 그리고 매 병렬 작업이 끝날 때마다 숨 돌릴 틈 없이 강박적으로 돌려야만 했던 인터페이스 계약 검증 프로세스의 실제 생생한 모습이 펼쳐진다.
