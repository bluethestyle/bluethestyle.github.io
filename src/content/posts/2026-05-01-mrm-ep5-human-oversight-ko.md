*"MRM 스레드" 5편. 에피소드 4에서 FRIA의 필수 요건 중 하나인 '인적 감독(Human Oversight)' 차원을 잠시 간단히 언급했다. 이번 편에서는 그 막연한 개념이 실제로 어떤 빠른 코드와 아키텍처로 구현되었는지를 낱낱이 펼쳐 보인다. EU AI 법안 제14조가 요구하는 인적 감독을, 처리 기약이 없는 일반적인 티켓 큐(Queue)가 아니라 즉각적으로 중단할 수 있는 API 엔드포인트로 살려낸 치열한 구조의 이야기다.*

## 공정성 사고(Incident)에 대응하는 골든타임

파트너 기관과의 실제 트래픽 수집은 2026년 4월 30일에야 비로소 시작되었으므로, 아직 프로덕션 환경에서 실제 고객의 항의로 촉발된 굵직한 공정성 사고(Incident)를 처리해 본 경험은 없다. 다만 이 시스템을 짤 때 우리의 설계 의도만큼은 매우 명시적이었다. **"당직(On-call) 엔지니어가 사고를 인지하고 즉각적으로 개입하는 판단의 골든타임이, 시스템의 정교한 탐지 능력 자체보다 훨씬 더 중요하다."** 그리고 아키텍처 전체가 바로 그 대응 창(Window)을 극단적으로 좁히는 방향에 집착하듯 맞춰져 있다.

이 시스템이 벼르고 있는 최악의 사고 시나리오를 하나 떠올려 보자. 공정성 모니터링 화면에 붉은색 경고등이 켜진다. 특정 보호군에 대한 위반, 예를 들어 65세 이상 고령층의 적금 상품 추천에서 이질적 영향(Disparate Impact) 비율이 곤두박질치며 금융소비자보호법 제17조 적합성 원칙의 마지노선 아래로 떨어져 버린 것이다. 

이때 전담 인력도 부족한 중소 규모의 ML 팀이 취할 수 있는 대응 경로는 현실적으로 세 가지뿐이다. 
(1) 며칠 뒤로 예정된 다음 정기 재학습 스케줄(보통 4–7일 뒤)까지 묵묵히 버틴다.
(2) 사내 인적 리뷰 시스템(Jira 등)의 백로그(Backlog)로 티켓을 밀어 넣고 누군가 배정되기를 기다린다 (한국 IT 운영 캘린더의 건강한 페이스라면 보통 12시간에서 24시간이 소요된다).
(3) **지금 당장, 시스템의 제어권을 가지고 직접 개입한다.**

앞의 두 가지 선택지는 규제 관점에서 절대 받아들일 수 없는 심각한 오답이다. 금융감독원의 사고 보고 가이드라인은 **"중대 사고 발생 시 24시간 이내 유선 보고, 72시간 이내 서면 보고"**를 엄격하게 강제하기 때문이다. 해결을 위해 5일을 기다리거나 티켓 처리를 위해 24시간을 허비하는 것은 사고에 대한 '대응'이 아니라, 감독 당국에 핑곗거리조차 댈 수 없는 '2차 사고'의 방조나 다름없다.

그렇다면 우리가 택한 세 번째, "지금 당장 개입한다"는 것은 실제로 어떤 모습일까? 전통적인 티켓 큐 기반 대응은 '긴급' 분류 후에도 업무 시간 내 분류(Triage) → 배정 → 처리 순서를 거쳐야 한다. 전체 사이클이 **분 단위가 아닌 시간–일 단위**로 측정되는 구조다. 우리의 아키텍처에서는 당직 엔지니어의 **API 호출 단 한 번**으로 개입이 완료된다. 이것은 예산의 문제가 아니라 설계 시점의 우선순위 선택이다.

## 관습적 조직 프로세스와 API 엔드포인트의 차이

EU AI 법안 제14조는 명시한다. *"고위험 AI 시스템은 이용자가 효과적으로 감독할 수 있는 수단을 반드시 제공해야 한다."* 흥미로운 점은 "효과적인 감독 수단"이 정확히 어떤 기술적 형태를 띠어야 하는지까지 규제가 세부적으로 규정하지는 않는다는 것이다. 일반적인 대응은 **조직 프로세스(Organization Process)** 형태다. *"리스크 관리 위원회에서 모니터링 지표를 정기 검토합니다"*, *"이상 수치 탐지 시 담당자에게 긴급 이메일이 발송됩니다"*, *"비상 상황에서는 인시던트 티켓을 생성해 처리합니다"* 같은 방식이다.

이 서면 프로세스들은 제14조 *문언상*의 요건을 충족한다. 다만 새벽 3시에 실제 사고가 발생했을 때의 작동 여부는 별개의 질문이다. 담당자가 긴급 이메일을 즉시 확인할 수 있는가? 티켓 큐의 선행 백로그는 어느 정도인가? 월례 위원회 일정상 3주 전 발생 사고의 검토가 지금 이루어지는 상황은 아닌가? 이런 변수들이 사후에 의문으로 남을 여지가 있다.

우리의 접근은 이 변수들을 구조적으로 줄이는 데 초점을 둔다. 감독 수단을 서면 프로세스가 아니라 **API 엔드포인트(Endpoint)**로 제공한다. '프로세스'가 아니라 즉각적인 **'호출(Call)'**이다. 호출은 실시간으로 기록되고, 몇 번을 호출해도 시스템 상태를 일관되게 보장(Idempotent)한다. 결재 큐의 선행 항목에 가려 지연될 여지가 없다.

## 킬 스위치 이전에 — 코드가 먼저 거부한다

이하 섹션들은 *사람이 어떻게 개입하는가* 에 대한 이야기다. 당직 엔지니어가 API 를 호출하고, tier 3 케이스가 인적 리뷰어에게 올라가고, 운영자가 force-promote 로 수동 승격을 결재한다. 그런데 이 모든 개입은 *사고가 표면화된 뒤* 의 개입이다. 그 앞단에, **사람이 판단하기 전에 코드가 먼저 후보를 제거하는 두 축**이 서빙 경로 안에 숨어 있다.

**기관 쪽 거부 — 적합성 필터.** 금융소비자보호법 제17조의 적합성 원칙은 "고객의 투자 성향과 상품의 위험 등급이 맞지 않으면 권유할 수 없다" 를 요구한다. 우리 파이프라인에서 이 요건은 문서로 있지 않고, 서빙 Lambda 안에 **실시간 필터** 로 박혀 있다. 각 고객의 위험 허용도와 각 후보 상품의 위험 등급을 매 요청마다 비교해서, 넘어가면 후보 목록에서 탈락. 65세 이상 고령층 고객에게 고위험 상품, 혹은 연소득 3,000만 원 미만 고객에게 자본시장 상품은 **hard cap** 으로 잠겨 있어 운영자 설정으로도 끌 수 없다. 이 필터를 통과한 후보만 ranking 과 Reason Generator 로 넘어간다.

**고객 쪽 거부 — 자동화 결정 거부권과 설명요구권.** 이 권리의 주된 법적 근거는 개인정보보호법(PIPA) 제37조의2 이다. 2024년 3월부터 시행된 이 조항은 자동화된 결정이 정보주체의 권리·의무에 *중대한 영향* 을 미치는 경우 정보주체가 그 결정을 **거부** 할 수 있고 **설명** 을 요구할 수 있도록 규정한다. 금융 상품 추천이 고객의 재무 의사결정에 미치는 영향을 고려할 때, 이 조항은 추천 시스템에 직접 걸린다. AI 기본법 제34조 (고영향 AI 사업자의 이용자 보호 방안) 도 같은 방향을 보조적으로 뒷받침한다.

두 근거 모두 권리 행사는 *즉시 반영* 되어야 한다. 우리 구현에서는 각 고객의 거부 의사와 프로파일링 관련 선호가 **DynamoDB 단일 로우** 로 저장되어 있고, 서빙 Lambda 가 매 요청마다 먼저 읽는다 (단일 자리수 ms). 고객이 모바일 앱에서 "AI 추천 받지 않음" 토글을 누르는 그 순간, DynamoDB 에 기록이 쓰이고, 다음 요청부터 해당 고객은 AI 추천 경로를 타지 않고 룰 기반 폴백(Layer 3) 으로 자동 분기한다. 서면 신청 → 며칠 처리 → 수동 플래그 세팅 같은 종래 구조가 가지는 *반영 지연* 이 구조적으로 사라진다.

두 필터의 공통점은 **사람의 API 호출이 필요 없다**는 것이다. 킬 스위치는 당직 엔지니어가 호출해야 돌아가고, HumanReviewQueue 는 Safety Gate 가 tier 를 올려야 돌아간다. 하지만 적합성 필터와 opt-out 필터는 *매 요청의 기본 경로 안* 에 들어 있다. `log_guardrail` 에 필터 결과가 기록되고 `log_data_access` 에 opt-out 상태 조회 이력이 남지만, 결정 자체는 사람 개입 없이 끝난다. 규제 대응 관점에서 이 층의 가치는 두 가지다 — *표면화 자체를 줄인다* (사고가 나기 전에 미리 걸러지므로 애초에 킬 스위치 대상이 되지 않음), 그리고 *고객 권리 행사의 지연을 없앤다* (법이 부여한 권리가 서면 처리 주기에 묶이지 않음).

이 두 필터 위에 비로소 뒤따르는 킬 스위치와 HumanReviewQueue 가 얹힌다. 즉 *기계가 먼저 거부 → 그래도 뚫고 나온 이상 신호에 대해 사람이 개입* 의 순서다.

## 킬 스위치(Kill Switch) — 단 한 번의 호출로 특정 태스크 숨통 끊기

가장 단순하면서도 확실한 API는 바로 킬 스위치(Kill Switch)다. 얽히고설킨 파이프라인 전체를 단숨에 셧다운시키거나, 13개의 태스크 중 비정상적인 특정 태스크 하나만 정밀하게 타격해 비활성화시키거나, 아니면 오염된 모델을 과거의 안전한 버전으로 즉각 롤백(Roll-back)시키는 일. 이 모든 조치가 각각 단 한 번의 API POST 호출로 깔끔하게 끝난다.

```json
POST /admin/kill-switch
{
  "scope": "task",
  "target": "churn_signal",
  "reason": "DI breach in age 60+ segment",
  "operator": "engineer_id"
}
```

당직 엔지니어가 이 한 줄의 API를 호출하는 순간, 시스템 내부에서는 엄격한 조치들이 릴레이로 터져 나간다.
- 지목된 타겟 태스크(churn_signal)의 프로덕션 서빙 경로가 즉각적으로 차단된다 (Lambda 환경 변수가 실시간으로 덮어씌워진다).
- 에피소드 3에서 구축한 뼈대인 log_operation 테이블에 event="kill_switch_fired"라는 영구적인 감사 엔트리가 깊게 새겨진다.
- 관련된 모든 슬랙(Slack) 채널과 PagerDuty에 동시에 시뻘건 알림이 터진다.
- OpsAgent가 알아서 이 상황을 정리해 사후 보고용 인시던트 티켓을 자동으로 깎아낸다.

이 구조가 서면 중심 조직 프로세스와 다른 지점은 명확하다.** 새벽 3시, 공정성 모니터가 경고를 울렸을 때, 당직 엔지니어가 티켓 큐를 거치지 않고 곧바로 위 API를 호출해 해당 경로를 중단시킨다. 문제 상황을 인지한 **'판단 시점'**과 밸브를 잠그는 **'실행 시점'**이 일치한다. "문제 인지와 처리 사이에 시간이 벌어졌다"는 사후 변명이 구조적으로 나올 수 없는 구성이다.

## HumanReviewQueue — 사람의 개입이 필요한 Tier 2/3 자동 에스컬레이션

킬 스위치가 "이미 문제가 터졌을 때" 사용하는 최후의 수단이라면, 그 전 단계 — 즉, **"조금 위험해 보이니 사람이 한 번 검토해야 한다"** 수준의 케이스들을 자동으로 분류해 인간 검토자에게 넘기는 거름망이 바로 HumanReviewQueue다.

우리의 큐 시스템은 철저하게 3개의 티어(Tier)로 나뉘어 굴러간다.

- **Tier 1 (안전 영역):** 묻지도 따지지도 않고 자동 승인. 모델의 신뢰도 지표(전문가 게이트의 엔트로피가 낮음)가 탄탄하고, 공정성 지표가 안전선을 여유 있게 방어하고 있으며, 금융 적합성 필터를 무사히 통과한 깔끔한 케이스다. 프로덕션 트래픽의 우수한 대다수가 이곳을 부드럽게 통과한다.
- **Tier 2 (인적 검토 필요):** 공정성 마진이 아슬아슬하게 임계치를 핥고 있거나, 모델의 신뢰도가 턱걸이 수준이거나, 또는 특별히 보호받아야 할 고령층/저소득층 고객이 연관된 위험한 케이스들이다. 즉시 차단되지는 않지만, 요주의 큐에 묵직하게 적재되어 반드시 업무일 기준 24시간 이내에 인간 리뷰어의 까다로운 사후 검토를 거쳐야만 한다.
- **Tier 3 (즉시 차단 및 즉각적 인적 개입):** 명백하게 선을 넘은 공정성 위반 시그널, LLM 특유의 위험한 환각(Hallucination) 징후 탐지, 혹은 자본시장법이나 금융소비자보호법을 정면으로 위반하는 규제 키워드(내부자 정보 누설, 과장 광고 제한어 등)가 노골적으로 등장한 위험한 케이스들이다. 이 오류들은 고객의 화면에 도달하기도 전에 즉시 차단(Block)되며, 인간 관리자가 직접 눈으로 보고 판정하기 전까지 엄격한 보류 상태로 격리된다.

여기서 핵심은 이 티어를 사람이 일일이 "음, 이건 좀 위험하니 Tier 2로 올리자"라고 낡은 방식으로 분류하지 않는다는 것이다. 위 세 가지 조건 중 단 하나라도 임계치를 건드리면, 티어는 **시스템에 의해 엄격하게 자동으로 격상(Escalation)**된다. 즉, 이 티어 분류 체계 자체가 거대한 '규칙 집합(Rule Set)'이며, 바로 이 날카로운 규칙 집합 자체가 담당 임원이나 리스크 관리 부서의 까다로운 정기 심사 대상이 되는 것이다.

## auto_promote=false — 프로덕션 환경의 타협 불가한 기본 태세

에피소드 2에서 우리는 챔피언-챌린저 승격 게이트의 우아한 구조를 자랑하며, "모든 조건을 통과하면 모델이 자동으로 승격된다"고 설명했다. 물론 **기술적으로는** 그렇다. 하지만 실제 펄떡이는 프로덕션(Production) 환경에서는 상황이 전혀 다르다. 프로덕션 환경의 설정(Config)에는 auto_promote=false라는 묵직한 자물쇠가 강제로 채워져 있다. 새로운 챌린저 모델이 제아무리 뛰어난 메트릭을 앞세워 깐깐한 관문들을 모조리 통과했다 한들, 시스템 혼자서 멋대로 자동 승격의 축포를 터뜨리지는 못한다. 운영자가 비장하게 쳐놓은 수동 개입(Force-promote Override) 버튼을 명시적으로 누르기 전까지, 기존의 챔피언 모델은 요지부동 자리를 지킨다.

이 귀찮은 자물쇠가 왜 굳이 필요한 걸까? **"모든 관문을 통과했으니 자동 승격해도 100% 안전하다"**라는 명제 자체가, 결국 기술적으로 세팅해 둔 그 관문(Gate)들이 세상의 모든 변수를 막아낼 수 있다는 부정확한 가정 위에 서 있기 때문이다. 현실 세계의 진흙탕 속에서는 바로 그 가정 자체마저도 매번 인간의 깐깐한 의심과 판단을 거쳐야만 한다. "성능 지표는 뛰어난데, 이 모델이 진짜로 오늘 당장 고객들에게 노출되어도 안전한가?"라는 묵직한 질문은 결코 메트릭 수치 하나로 퉁칠 수 없다. 적어도 2026년 상반기 기준, 우리 팀은 새로 올라오는 모든 챌린저 모델의 배포 결정을 반드시 인간의 **수동 사인오프(Manual Sign-off)**를 거쳐야만 게이트가 열리도록 꽉 틀어막고 있다.

이 고집스러운 운영 태세(Posture)가 바로 EU AI 법안 제14조가 지속적으로 중점적으로 다루는 **"의미 있는 인적 감독(Meaningful Human Oversight)"** 조항과 성공적으로 톱니바퀴처럼 맞물린다. 메트릭만 보고 시스템이 알아서 밀어붙이는 자동 승격은 철저히 "기계의 결정(The system decides)"에 불과하지만, 우리가 고집하는 수동 승격은 분명한 "인간의 묵직한 결단(A human decides)"이기 때문이다. 실제 고객 데이터 환경에서 시스템이 성공적으로 길들여지고 안정화되기 전까지는, 이 피곤하지만 든든한 후자의 태세가 규제적으로도, 윤리적으로도 매우 옳다는 것이 우리의 결론이다.

## Layer 4 — 선택적 도입이 가능한 인적 폴백(Human Fallback)

고객에게 추천 결과가 서빙되는 치열한 경로 위에는 이미 3계층의 든든한 폴백(Fallback) 라우터가 버티고 있다.

- **Layer 1:** 묵직한 PLE 모델의 정수를 가볍게 압축해 낸 LGBM 증류(Distillation) 모델 (99% 이상의 우수한 일반 트래픽이 이 쾌적한 고속도로를 탄다).
- **Layer 2:** 만약 Layer 1의 증류 모델이 연산에 실패하거나 뻗어버릴 경우, 당황하지 않고 오리지널 LGBM이 직접 원본 데이터를 학습하여 예측을 뱉어내는 1차 방어선.
- **Layer 3:** AI 모델 전체가 시스템 전체 오류 상황에서도, 사전에 하드코딩해 둔 안전하고 보수적인 '금융 DNA 비즈니스 룰(Rule)'을 통해 억지로라도 추천을 짜내는 최후의 보루.

그런데 2026년 상반기, 여기에 매우 아날로그적인 방어선 하나가 조용히 추가되었다. 바로 **Layer 4 — 인적 폴백(Human Fallback)**이다. 이 레이어는 설정(Config) 파일에서 tier-3 human-fallback 플래그를 명시적으로 켜두었을 때만 자연스럽게 활성화된다. 만약 Layer 1, 2, 3이 내놓은 결과물조차 모두 모델 신뢰도(Confidence) 임계치를 밑도는 그야말로 대혼돈의 상황이 닥치면, 시스템은 부정확한 오류를 지어내 고객을 속이는 대신 추천 생성 자체를 **깔끔하게 포기**해 버린다. 그리고 고객의 화면에 정중하게 메시지를 띄운다. *"현재 고객님께 가장 적합한 상품을 분석 중입니다. 상세한 안내를 위해 전문 상담 직원에게 즉시 연결해 드리겠습니다."*

이 훌륭한 Layer 4를 굳이 기본값(Default)으로 강제하지 않고 선택적(Opt-in)으로 남겨둔 이유는 지극히 현실적이다. 모든 금융기관이 이런 골치 아픈 예외 케이스에 무작정 값비싼 인적 콜센터 리소스를 콸콸 쏟아부을 수 있는 여력이 되는 것은 아니기 때문이다. 선택의 문은 활짝 열어두되 기본값은 조용히 꺼둔다. 이 기능을 자신 있게 켠 기관은 반드시 뒷단에 무거운 인적 리뷰 큐(Human Review Queue) 조직을 별도로 운영해야만 하는 묵직한 비용을 감당해야 한다.

## 제14조의 요구에 죽은 문서가 아니라 생생한 행동으로 답하다

금융감독원 감사관이 "EU AI 법안 제14조(인적 감독)를 실제로 준수하고 있다는 증거를 제시해 달라"고 요청하는 상황을 상정해 보자. 우리는 서류 뭉치를 구성할 필요 없이, 데이터베이스에서 다음과 같은 로그를 곧바로 조회하여 제출할 수 있다.

- 당직 엔지니어가 절박하게 킬 스위치 API를 실행했던 호출 이력 → log_operation 테이블에 선명하게 각인되어 있다.
- 요주의 케이스들이 HumanReviewQueue의 Tier 2와 Tier 3로 전달되어 인간의 심사를 받았던 치열한 처리 이력 → 전용 감사 테이블에 차곡차곡 쌓여 있다.
- 챔피언 모델의 교체를 두고 인간 운영자가 어려운 고민 끝에 승인을 반려했거나 강제로 밀어붙였던 수동 사인오프 이력 → log_model_promotion 테이블의 trigger=manual 엔트리에 박혀 있다.
- Layer 4 기능이 켜져 있던 기관에서, 기계가 항복을 선언하고 인간 직원에게 고객을 넘겼던 인적 폴백 경로의 생생한 호출 횟수 → 별도의 집계 테이블에 숫자로 남아 있다.

이 기록들은 사후에 작성된 서술이 아니라, **실제 API 호출과 이벤트 발생 시 시스템이 자동으로 기록한 부산물**이다. "인적 감독 프로세스를 매뉴얼로 *갖추고 있습니다*"라는 답변과, "지난 12개월 동안 킬 스위치를 N번 *실행했습니다*"라는 로그 기반 답변은 감사 대응력(Auditability) 측면에서 다른 차원의 증거다.

## 시스템이 끝내 대체할 수 없는 사람의 영역

세련된 API 엔드포인트를 제공하는 것과, **누군가가 책임을 지고 그 API를 실제로 실행하는 것**은 완전히 차원이 다른 문제다. 제아무리 성공적인 킬 스위치 API를 만들어 두었더라도, 새벽 3시에 잠을 쫓으며 당직을 서던 엔지니어가 모니터에 번쩍이는 붉은색 공정성 알람을 보고도 "에이, 아침에 출근해서 보지 뭐"라며 묵살해 버린다면 그 훌륭한 코드는 한낱 무용지물에 불과하다.

그러므로 우리가 입이 닳도록 자랑한 "API 기반의 인적 감독(Human Oversight)" 아키텍처는, 인간을 시스템에서 배제하려는 기계적 자동화의 논증이 절대 아니다. 오히려 거추장스러운 프로세스를 걷어내어 **"인간이 개입하는 데 드는 비용과 장벽을 극단적으로 낮추어 주는"** 치열한 인간 중심의 논증이다. 몇 단계를 거쳐 결재를 받아야 하는 지라(Jira) 티켓 큐를 통과하는 대신, 당직 엔지니어의 명확한 판단이 24시간의 일반적인 대기 없이 곧바로 **생생한 행동**으로 치환되도록 설계한 것이다. 그렇게 낡은 프로세스의 군살을 모두 발라냈음에도 불구하고, 결국 마지막 스위치를 누를 것인지 말 것인지를 결정하는 **'판단 그 자체의 질'**은 여전히, 그리고 영원히 고독한 인간의 몫으로 남는다.

이 아키텍처 위에서 담당 임원이나 리스크 관리 부서의 검토 방식도 바뀐다. 월례 위원회에서 실무자가 정리한 요약 보고서를 검토하는 흐름에서, API 호출 로그를 직접 분석하는 흐름으로 이동한다. *"지난주 새벽 3시에 엔지니어가 킬 스위치를 당긴 판단은 과연 합당했는가?"*, *"Tier 2와 Tier 3로 분류되어 인간에게 넘어오는 요주의 케이스들의 비율이 지금 건강한 수준인가, 아니면 시스템의 거름망이 헐거워진 것인가?"*, *"최근 챌린저 모델들에 대해 수동 승격 거부를 때린 건수가 너무 과도하게 많은 것은 아닌가?"*

**결국 관리 감독 조직의 시선이, 예쁘게 포장된 '사후 리포트'에서 시스템이 뱉어낸 '날것의 호출 로그'로 완전히 이동한 것이다.** 이것이 바로 규제가 요구하는 인적 감독이 문서 더미를 탈출해 실제 코드 위에서 살아 숨 쉬게 된 가장 성공적인 증거다.

## 다음 편

이어지는 **에피소드 6**에서는, EU AI 법안 제14조(인적 감독)의 사촌 격이자 모든 규제의 뜨거운 감자인 **'공정성(Fairness)'**을 낱낱이 해부한다. 이질적 영향(Disparate Impact), 통계적 패리티 차이(Statistical Parity), 균등 기회 차이(Equal Opportunity) 같은 골치 아픈 공정성 지표들을, 안전하게 격리된 한 줌의 검증 데이터 셋에서 그럴듯하게 한 번 측정하고 치워버리는 것이 아니라 **거칠게 쏟아지는 프로덕션 스트림 위에서 매 예측마다 실시간으로** 수행하며 계산해 내는 과격한 아키텍처를 공개한다. 아울러 에피소드 4에서 잠시 스쳐 지나갔던 '반사실적 챔피언-챌린저(Counterfactual Champion-Challenger)' 구조가, 실시간 공정성 판단의 사각지대에서 도대체 어떤 결정적인 역할을 해내는지 그 치열한 원리를 파고든다.
