*"3개월 개발기" 3편. Ep 2 에서 3인 × AI 팀이 병렬로 모듈을 만들었다고
썼다. 이번 편은 그 병렬이 통합 지점에서 깨지지 않게 한 장치들 —
CLAUDE.md 기본 규약, 메모리 뱅크, 인터페이스 계약 검증 — 의 구체적
모습.*

## 3일 뒤 통합의 풍경

프로젝트 초기 어느 목요일. 3개 팀이 화요일부터 각자 AI 에이전트에
작업을 맡겼다. PM 팀은 모델 구성 스키마를, 엔지니어 1 팀은 피처
스키마를 생성하는 전처리 파이프라인을, 엔지니어 2 팀은 그 둘을
읽어 모델을 구성하는 학습 스크립트 초안을.

3명이 목요일 오후에 모여 통합을 시도했다. 결과 — 30분 만에 런타임
에러 5개. 모두 *같은 원인* 의 변형이었다. PM 팀의 스키마는 피처
그룹 경계를 어떤 키 이름으로 저장했다. 엔지니어 1 팀의 파이프라인은
다른 이름으로 저장했다. 엔지니어 2 팀의 학습 스크립트는 또 다른
이름으로 읽었다. 셋 다 *그럴듯한* 이름이었다.

이게 병렬 AI 작업의 고유 실패 모드다. 각 에이전트는 자기 맥락에서
가장 합리적인 이름을 선택한다. 세 맥락의 합리적 이름 셋이 일치할
확률은 0 에 수렴한다.

해결책은 두 가지가 있다. 하나 — 병렬을 포기하고 직렬로 돌린다 (3일
걸릴 작업이 9일). 둘 — 병렬이 수렴하도록 *사전 장치* 를 깐다. 우리가
택한 둘째 접근이 이번 편의 주제다.

## CLAUDE.md 기본 규약 — 코드보다 먼저

프로젝트 루트의 `CLAUDE.md` 는 Phase C (Cursor 기반 환경 정비)
단계에서 *코드 한 줄 작성되기 전에* 쓰였다. 순서가 의도적이었음을
Ep 2 에서 짚었고, 이 파일이 프로젝트의 성격을 가장 많이 결정했다.

현재 버전은 여러 번 확장되어 열다섯 개 넘는 섹션으로 커졌지만,
초기에 박아넣은 네 가지가 가장 중요했다:

**Config-driven 원칙.** 모든 파라미터는 YAML config 에서 읽는다.
컬럼명·경계값·시나리오 목록·AWS 상수를 코드에 하드코딩하지 않는다.
왜 이게 *기본급 규약* 인가 — AI 에이전트는 "그럴듯한 상수" 를 코드에
박아넣는 데 너무 능숙하다. `batch_size=5632` 같은 숫자가 주석 없이
학습 스크립트에 박히면 3주 뒤 왜 그 값인지 아무도 모른다. Config 에
강제 위치시키면 *문서화와 변경 가능성이 한 번에 해결* 된다.

**관심사 분리.** 어댑터는 raw → standardized DataFrame 만. 파이프라인
러너는 전처리·피처 생성·레이블 파생·정규화·텐서 저장 전부. 학습
스크립트는 training-ready 데이터 로드 → 모델 빌드 → 학습만. 경계가
모호해지면 500줄을 넘는다. 넘으면 관심사 분리가 안 된 것으로 판정.

위 목요일 통합 사고의 근원이 여기 있었다. 엔지니어 2 팀의 학습
스크립트에 전처리 일부가 인라인으로 박혀 있었고, 그 전처리가 엔지니어
1 팀이 Phase 0 에서 만든 스키마와 다르게 동작했다. 분리 원칙을
사후에 강제하면서 230줄의 전처리가 Phase 0 으로 이관됐다.

**데이터 리키지 방지.** 스케일러는 train split 에서만 fit. 시간적
split 에 gap_days 필수. 학습 전 LeakageValidator 강제 호출. 이
조항은 Ep 5 (데이터 무결성 사냥) 에서 자세히 다룬다.

**완료 보고 4단계.** "완료" 라고 말하려면 — 컴파일 검증 · 인터페이스
계약 검증 (아래 별도 절) · 하드코딩 스캔 · 관심사 분리 점검. 네
가지 전부 통과 후에만 "완료". 에이전트가 "다 했어요" 라고 하는데
사실은 한두 단계만 끝났을 때 파이프라인이 조용히 깨진다.

## 메모리 뱅크 — 수동에서 auto-memory 로

`CLAUDE.md` 가 프로젝트 규약집이라면, 세션 사이의 연속성을 유지하는
장치가 메모리 뱅크다. 이 부분의 실현 방법이 프로젝트 내내 *두 번* 바뀌었다.

**초기 접근 — 수동 8 파일 구조.** 2025년 말 ~ 2026년 초에는 Claude
Code 의 세션간 기억이 제한적이었다. 그래서 `.claude/memory-bank/`
디렉토리에 직접 8개 파일을 두고 매 세션 시작에 핵심 파일을 에이전트
에게 읽히는 방식으로 대응했다 — 프로젝트 요약, 지금 작업 중인 맥락,
이정표 체크리스트, 기술 스택 노트, 제품 맥락, 반복 패턴, 태스크
상태, 스타일 가이드. 파일을 여러 개로 쪼갠 건 경험적 발견이었다 —
한 파일에 모두 담으면 첫 세션에서 에이전트가 overwhelming 해서 핵심
을 놓친다는 것을 몇 주 안에 배우고 난 뒤.

도구를 여러 개 병행하던 시기에는 `.claude/RULES.md` 와 `.cursorrules`
를 canonical source / derived copy 관계로 유지하는 동기화 스크립트
도 있었다. 여러 도구가 서로 다른 규칙을 따르면 출력이 어긋나서 파이프
라인 통합 지점에 또 깨짐이 쌓이기 때문.

**2026년 중반 — Claude Code auto-memory 도입 이후.** 2026-04 에
Claude Code 가 v2.1.59+ 에서 auto-memory 를 정식 도입하면서 세션
간 지식이 자동으로 누적되기 시작했다. 빌드 명령, 디버깅 통찰, 아키텍처
노트, 코드 스타일 선호, 워크플로 습관 — 이전에 수동 파일로 관리하던
대부분이 에이전트가 알아서 기억하는 영역으로 이동했다.

그래서 현재는 수동 8 파일 중 대부분을 쓰지 않는다. 남은 건 두 개:
*프로젝트 목표·제약* 을 명시한 최상위 요약 한 파일과, *지금 진행 중인
태스크 상태* — 자동 기억에 의존하기엔 너무 세밀히 바뀌는 것. 나머지는
auto-memory 가 대체한다. `.cursorrules` 동기화도 마찬가지로 단계적
으로 접어 뒀다 — Claude Code 쪽 도구 일원화가 진행되면서 병행 유지의
이득이 줄었기 때문.

이 이행 과정 자체가 한 가지 교훈을 남겼다 — *도구가 약했을 때 만든
가드레일을 도구가 강해지고 나서도 관성으로 유지하지 않는다*. 수동
메모리 뱅크는 2026년 초엔 반드시 필요했지만, 지금은 이미 구식이
된 패턴이다.

## 인터페이스 계약 검증 — 병렬 작업 후의 필수 루틴

위에서 언급한 목요일 통합 사고 — 같은 개념이 세 팀에서 서로 다른 키
이름으로 저장·소비됐던 문제 — 를 다시는 겪지 않도록, 모든 병렬 작업
세션 종료 시 다음 루틴이 돌아간다:

1. 수정된 각 파일의 저장 함수가 어떤 키로 dict 를 저장하는지 추출
2. 같은 브랜치의 다른 파일의 로드 함수가 어떤 키로 읽는지 추출
3. 두 집합의 diff 를 확인 — 저장만 하고 안 읽히는 키, 읽으려는데
   없는 키, 양쪽에 있지만 이름이 다른 키

이 루틴을 AI 에이전트에게 위임한다. 병렬 에이전트 2-3개가 작업을
끝내면, *네 번째 에이전트* 가 인터페이스 계약 검증 전용으로 호출된다.
병렬 에이전트들은 각자의 맥락에서 "그럴듯한" 이름을 선택하지만, 검증
에이전트는 *전체를 조망* 하는 맥락에서 불일치를 식별한다.

이 네 번째 에이전트가 없으면 목요일 통합 사고가 반복되어 디버깅이
매주 3시간씩 소모된다. 있으면 그 3시간이 *20분* 으로 줄어든다 —
불일치가 CI 단계 훨씬 전에 발견되기 때문.

## 가드레일이 AI 협업에만 쓰이는 건 아니다

이 세 장치 — CLAUDE.md 규약, 세션간 맥락 유지, 계약 검증 — 는 AI
에이전트 때문에 만들어진 것처럼 들리지만, 실은 *3인 팀 협업에도 그대로
적용* 된다. 3명이 병렬로 작업할 때의 문제 (이름 불일치, 맥락 유실,
규칙 drift) 는 AI 3명 병렬의 문제와 동형이다. 우리가 이걸 AI 협업
맥락에서 먼저 체계화했을 뿐, 원칙은 인간 팀 협업의 오래된 지혜와
같다.

이게 한국 금융권 중소 규모 팀 (3-5명 규모) 에 이 구조가 이식 가능한
이유다. Claude Code 구독이 없어도 CLAUDE.md 수준의 프로젝트 규약집,
공유 맥락 문서, 매 통합 전 키 매칭 검증 루틴은 그대로 설치된다.
AI 가 있으면 더 빠르고 없어도 작동한다.
